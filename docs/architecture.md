## 2. 核心设计理念

### 2.1 大小模型协同

本系统的核心创新在于**大小模型协同**架构：

| 维度 | 小模型 | 大模型 |
|------|--------|--------|
| **部署位置** | 边缘端/本地 | 云端 |
| **响应速度** | 毫秒级 | 秒级 |
| **资源消耗** | 低 | 高 |
| **推理能力** | 模式匹配 | 深度推理 |
| **应用场景** | 数据处理、异常检测 | 影响评估、报告生成 |

**协同优势**：
- 小模型实时处理海量遥测数据，过滤正常数据
- 仅将异常数据传递给大模型，减少调用成本
- 大模型专注于需要深度推理的任务
- 整体系统既高效又智能

### 2.2 多Agent架构

使用LangGraph实现的有状态工作流：

- **数据处理Agent**：负责数据清洗、特征工程、统计分析
- **异常检测Agent**：多策略融合检测（规则+统计+ML）
- **影响评估Agent**：LLM+RAG驱动的智能评估
- **报告生成Agent**：自动化运维报告生成
- **条件路由**：无异常时跳过评估，节省资源

### 2.3 RAG增强

将检索增强生成技术应用于运维场景：

1. **知识库构建**：将运维手册、技术规格、最佳实践等文档向量化
2. **语义检索**：基于FAISS的高效向量检索
3. **增强生成**：检索结果作为上下文，辅助LLM生成更准确的评估

## 3. 模块详解

### 3.1 数据处理模块 (`src/models/data_processor.py`)

**功能**：
- 遥测数据验证和清洗
- 派生特征计算（电池损耗率、速度变化、电机方差等）
- 数据质量评估
- 编队统计分析

**关键指标**：
- 支持10+维度的遥测数据处理
- 维护每架无人机最近100条历史记录
- 实时计算5+个派生特征

### 3.2 异常检测模块 (`src/models/anomaly_detector.py`)

**检测策略**：
1. **规则引擎**：基于阈值的快速检测（温度、振动、电量、信号）
2. **电机平衡检测**：RPM方差分析，检测机械故障
3. **趋势分析**：基于派生特征的趋势异常检测
4. **Isolation Forest**：无监督机器学习异常检测

**输出**：
- 异常类型分类（7种类型）
- 严重等级（正常/警告/危急）
- 置信度评分
- 处置建议

### 3.3 RAG模块 (`src/rag/`)

**组件**：
- `document_loader.py`：知识库文档加载和分块
- `vector_store.py`：FAISS向量存储管理
- `retriever.py`：RAG检索链实现

**知识库内容**：
- 无人机运维手册（故障类型、处置流程）
- 技术规格文档（硬件参数、性能指标）
- AIOps最佳实践（架构设计、评估标准）

### 3.4 LangGraph工作流 (`src/agents/orchestrator.py`)

**工作流节点**：
```python
data_processing -> anomaly_detection -> [条件路由]
                                            ├── 有异常 -> impact_assessment -> report_generation -> END
                                            └── 无异常 -> simple_report -> END
```

**特性**：
- TypedDict定义的类型安全状态
- 条件路由优化资源使用
- 错误传播和处理
- 完整的时间跟踪

## 4. 数据流

```
遥测数据(JSON) 
    │
    ▼
[DataProcessor.process_telemetry]
    │ 清洗 + 特征工程
    ▼
处理后数据(含派生特征)
    │
    ▼
[AnomalyDetector.detect]
    │ 规则 + 统计 + ML
    ▼
异常检测结果(AnomalyResult)
    │
    ├── 有异常 ──▶ [AssessmentAgent.assess]
    │                │ LLM + RAG检索
    │                ▼
    │           影响评估报告
    │                │
    │                ▼
    │           [ReportAgent.generate_report]
    │                │ 完整运维报告
    │                ▼
    │           最终报告(Markdown)
    │
    └── 无异常 ──▶ 简报生成
                    │
                    ▼
                简要状态报告
```

## 5. API设计

| 端点 | 方法 | 描述 |
|------|------|------|
| `/health` | GET | 系统健康检查 |
| `/api/v1/analysis/full` | POST | 运行完整分析流水线 |
| `/api/v1/analysis/detect` | POST | 仅运行异常检测 |
| `/api/v1/analysis/sample` | GET | 获取样本数据 |
| `/api/v1/analysis/sample/analyze` | POST | 分析样本数据 |
| `/api/v1/rag/query` | POST | RAG知识问答 |
| `/api/v1/rag/assess` | POST | RAG影响评估 |
| `/api/v1/rag/status` | GET | RAG系统状态 |
