## Q1: 请介绍一下这个项目的整体架构，为什么选择大小模型协同的设计？

### 参考答案：

本项目是一个**无人机集群异常检测与影响评估的AI智能运维平台**，采用大小模型协同的多Agent架构。

**整体架构**分为三层：
1. **小模型层**：部署在边缘端，包括数据处理器和异常检测器。数据处理器负责遥测数据清洗和特征工程，异常检测器使用规则引擎+统计分析+Isolation Forest进行多策略异常检测。
2. **大模型层**：部署在云端，使用GPT-4o等LLM负责影响评估和报告生成。通过RAG技术结合运维知识库，提供知识增强的评估结果。
3. **编排层**：使用LangGraph实现有状态的多Agent工作流，包含条件路由逻辑。

**选择大小模型协同的原因**：
- **效率**：无人机集群产生海量遥测数据，小模型可以毫秒级处理，过滤掉大量正常数据
- **成本**：仅将异常数据传递给大模型，避免不必要的API调用，在我们的测试中大模型调用减少了约90%
- **可靠性**：小模型不依赖网络，即使云端不可用也能保证基础检测能力
- **互补性**：小模型擅长模式匹配和快速检测，大模型擅长深度推理和自然语言生成

## Q2: 你的异常检测模块具体是怎么实现的？准确率如何？

### 参考答案：

异常检测采用**多策略融合**的方案，包含三个层次：

**1. 规则引擎检测**（第一层）
- 基于预设阈值的快速检测，覆盖温度、振动、电量、信号强度等核心指标
- 优点：低延迟（<1ms）、可解释性强、零误判（针对阈值内的明确异常）
- 例如：温度>80°C、电量<10%、信号强度<-90dBm

**2. 统计分析检测**（第二层）
- 电机RPM方差分析：检测机械不平衡（变异系数>15%判定为异常）
- 趋势分析：基于历史数据的派生特征（电池损耗率、高度稳定性、速度突变）
- 优点：能发现渐变型异常和组合型异常

**3. Isolation Forest**（第三层）
- 无监督机器学习算法，检测高维空间中的异常点
- 使用正常飞行数据训练，contamination参数设为0.05
- 优点：能发现规则和统计方法难以覆盖的未知异常模式

**性能指标**：
- 在注入10%异常的500条模拟数据测试中：
  - 检测准确率：约95%+（规则层面对明确异常接近100%）
  - 异常类型覆盖：7种（温度、振动、电池、信号、电机、高度、速度）
  - 检测延迟：单条记录<5ms
  - 支持批量检测和实时流处理

## Q3: RAG在这个项目中具体是怎么使用的？为什么要用RAG而不是直接让LLM回答？

### 参考答案：

**RAG的使用场景**有两个：
1. **运维知识问答**：运维人员可以通过自然语言查询运维手册、技术规格等信息
2. **异常影响评估增强**：检测到异常后，RAG从知识库中检索相关的故障处理方案和安全标准，辅助LLM进行更准确的影响评估

**RAG实现细节**：
- **知识库构建**：将运维手册、技术规格、最佳实践等Markdown文档使用RecursiveCharacterTextSplitter分块（chunk_size=500, overlap=100）
- **向量化存储**：使用OpenAI text-embedding-3-small模型生成嵌入，存储在FAISS向量数据库中
- **检索增强**：查询时先检索top-K相关文档片段，作为上下文传递给LLM

**为什么用RAG而不是纯LLM**：
1. **准确性**：LLM的训练数据可能不包含我们特定的无人机型号参数和运维规程，RAG确保回答基于我们自己的知识库
2. **时效性**：运维规程会更新，RAG只需更新知识库文档，不需要重新训练模型
3. **可追溯性**：RAG返回引用来源，运维人员可以验证信息的准确性
4. **幻觉控制**：通过限制LLM只能基于检索到的内容回答，减少幻觉（hallucination）

## Q4: LangGraph在项目中起到什么作用？和直接串行调用有什么区别？

### 参考答案：

LangGraph在本项目中实现了**有状态的多Agent工作流编排**。

**具体实现**：
```
data_processing → anomaly_detection → [条件路由] 
                                        ├── 有异常 → impact_assessment → report_generation → END
                                        └── 无异常 → simple_report → END
```

**相比直接串行调用的优势**：
1. **条件路由**：当没有检测到异常时，跳过LLM影响评估环节，节省约60%的处理时间和100%的LLM调用成本
2. **状态管理**：使用TypedDict定义的全局状态在节点间传递，每个Agent只需要关注自己的输入输出
3. **错误处理**：每个节点独立捕获错误，不影响其他节点执行，错误信息通过状态传递
4. **可扩展性**：新增Agent只需添加节点和边，不需要修改现有代码
5. **可视化**：LangGraph支持工作流可视化，方便调试和理解
6. **可重放性**：有状态的设计使工作流可以从任意节点重新执行

**实际效果**：
- 正常数据处理：平均50ms（跳过LLM调用）
- 异常数据处理：平均2-5秒（包含LLM调用）
- 在90%正常数据的典型场景下，系统整体效率提升显著

## Q5: 你的项目中如何保证系统的可靠性和容错性？

### 参考答案：

系统在多个层面实现了可靠性保障：

**1. 分层容错**
- 小模型层：不依赖外部API，纯本地计算，即使网络断开也能正常工作
- 大模型层：实现了`assess_without_llm`和`generate_report_without_llm`两个fallback方法，当LLM不可用时自动切换到规则/模板模式
- RAG层：初始化失败时不影响其他功能，系统降级运行

**2. 数据质量保障**
- `DataProcessor`为每条记录计算数据质量评分
- 缺失字段检测和记录
- 异常值范围校验
- 历史记录保留最近100条，防止内存溢出

**3. Pipeline错误传播**
- LangGraph工作流中每个节点独立捕获异常
- 错误信息通过状态的`errors`字段传递
- 不会因为单个节点失败而导致整个pipeline崩溃

**4. API层保障**
- FastAPI的Pydantic模型验证输入数据
- 完善的HTTP错误码和错误消息
- CORS配置支持跨域访问

## Q7: 你在项目中遇到过什么技术难点？是如何解决的？

### 参考答案：

**难点1：LangGraph状态类型设计**
- 问题：多Agent之间需要传递复杂的嵌套数据结构，TypedDict在类型检查方面不够灵活
- 解决：使用TypedDict定义核心状态字段，复杂数据使用dict类型存储，在每个节点内部进行验证

**难点2：RAG检索质量优化**
- 问题：中文文档分块时容易在语义不完整的地方切断
- 解决：自定义RecursiveCharacterTextSplitter的分隔符优先级，将中文标点（"。"）加入分隔符列表，确保按语义边界分块

**难点3：大小模型协同的可靠性**
- 问题：大模型（云端API）不稳定时会影响整个系统
- 解决：实现了双模式设计——每个大模型Agent都有对应的fallback方法（如`assess_without_llm`），当LLM不可用时自动降级到规则/模板模式

**难点4：异常检测的多策略融合**
- 问题：不同检测策略可能对同一记录给出矛盾的结果
- 解决：设计了分层检测策略，规则引擎作为基线，统计分析补充趋势异常，ML方法发现未知异常，最终结果取并集并按严重性排序

## Q9: 如果面对面试官追问"这个项目的创新点是什么"，你会怎么回答？

### 参考答案：

**创新点1：大小模型协同的AIOps架构**
- 不同于纯LLM方案或纯ML方案，本项目将两者结合
- 小模型负责高频低延迟的数据处理和异常检测
- 大模型专注于需要深度推理的影响评估和报告生成
- 这种设计在实际部署中具有很好的成本效益比

**创新点2：RAG增强的运维影响评估**
- 将RAG应用于工业运维场景，而非通常的文档问答
- 影响评估时自动检索相关的技术规格和处置规程
- 评估结果既有LLM的推理能力，又有知识库的准确性保障

**创新点3：LangGraph条件路由优化**
- 工作流不是简单的串行执行，而是根据异常检测结果动态路由
- 正常情况下跳过LLM调用，显著降低成本和延迟
- 体现了"按需使用AI"的设计理念

**创新点4：多策略融合的异常检测**
- 规则引擎 + 统计分析 + Isolation Forest三层检测
- 不同策略互补，提高检测的覆盖率和准确性
- 每种策略都有明确的适用场景和优缺点分析

## Q10: 你对RAG技术的理解是什么？RAG有什么局限性？

### 参考答案：

**RAG的本质**：
RAG（Retrieval-Augmented Generation）是一种将信息检索与文本生成结合的技术范式。核心思想是"先检索，再生成"——将用户查询与知识库中的文档进行语义匹配，将最相关的文档片段作为上下文传递给LLM，让LLM基于这些上下文生成回答。

**RAG的工作流程**：
1. 文档分块 → 向量化 → 存储到向量数据库
2. 用户查询 → 查询向量化 → 相似性检索 → 获取top-K文档
3. 原始查询 + 检索文档 → LLM → 生成回答

**RAG的局限性**：
1. **检索质量依赖**：如果检索不到相关文档，生成质量会下降。解决方案：优化分块策略、使用混合检索（关键词+语义）
2. **上下文窗口限制**：检索的文档数量受LLM上下文窗口限制。解决方案：使用压缩重排序（re-ranking）
3. **语义理解局限**：嵌入模型可能无法准确捕捉专业领域的语义。解决方案：使用领域微调的嵌入模型
4. **实时性**：知识库更新需要重新向量化。解决方案：增量索引更新
5. **多跳推理**：RAG不擅长需要跨多个文档进行推理的复杂问题。解决方案：结合思维链（Chain of Thought）
